# MMMResidenceChunkBridge 未完成项与优化建议

## 0. 文档维护要求

每次修改插件后必须同步更新 `README.md` 中的“更新记录”。

版本号规则：

- `x`：重大架构变更
- `y`：新增功能
- `z`：Bug 修复

版本号需要同步更新：

- `pom.xml`
- `src/main/resources/plugin.yml`
- `README.md`

## 1. 当前未完成的内容

## 1.1 创建领地名称输入体验还不完整

当前创建领地时：

- 菜单点击后进入粒子预览
- 聊天输入 `确认` 才会创建

但还没有做到：

- 在创建流程里让玩家直接输入领地显示名
- 让“输入名称 -> 预览 -> 确认”形成完整闭环

现在默认还是：

- 菜单创建时使用自动显示名，如 `领地1`
- 或使用命令 `/mmmland create [显示名]`

这部分后续可以改成更完整的会话式流程。

## 1.2 扩张和缩小还不是确认式流程

当前扩张和缩小是：

- 菜单选择方向
- 菜单选择区块数
- 显示预览
- 直接执行

这意味着：

- 玩家虽然能看到预览
- 但没有像创建领地那样再进行一次确认

如果后续想统一体验，可以改成：

- 预览后聊天确认
- 或预览后再次点击确认按钮

## 1.3 重命名仍依赖命令

当前领地重命名不是完整菜单流程，而是提示玩家执行命令：

```text
/mmmland rename <旧显示名> <新显示名>
```

这会带来两个问题：

- 交互割裂
- 对普通玩家不够直观

## 1.4 旧 Residence 领地没有自动接管机制

当前本插件只管理：

- 由本插件创建
- 并写入 `claims.yml` 的领地

未完成的部分是：

- 自动识别历史 Residence 领地
- 将历史领地纳入本插件列表
- 导入已有领地为“托管领地”

## 1.5 外部菜单入口整合已清理

当前已经移除旧的 Invero 入口自动导出，菜单由插件内置 Bukkit GUI 负责。

后续如果需要接入服务器主菜单，建议只保留一个外部按钮调用：

- `/mmmland menu`
- `/mmmland select`

## 1.6 管理员功能还不完整

目前管理员能力比较弱，主要只是部分所有权绕过。

还缺少常见管理功能：

- 查看所有玩家领地
- 强制删除他人领地
- 强制修改他人领地显示名
- 查询领地价格计算结果
- 重载配置
- 导入/修复托管数据

## 1.7 数据自检与修复能力不足

当前 `claims.yml` 主要是正常读写，没有做更强的数据自检。

还缺少：

- 启动时检查记录对应的 `Residence` 领地是否存在
- 发现缺失时自动清理无效记录
- 发现重名显示名时给出修复工具

## 2. 当前可以继续优化的方向

## 2.1 把创建流程做成完整会话系统

建议后续新增一个“会话状态管理器”，统一管理：

- 创建会话
- 扩张会话
- 缩小会话
- 删除确认会话
- 重命名输入会话

这样比把状态分散在 `GuiService` 更清晰。

建议流程可改成：

1. 菜单点击创建
2. 关闭菜单
3. 粒子预览
4. 聊天输入名称
5. 聊天输入确认
6. 完成创建

这样会更自然。

## 2.2 粒子预览效果优化

当前粒子预览已经可用，但还比较基础。

可以继续优化为：

- 不同状态使用不同颜色
- 创建预览、扩张预览、缩小预览用不同粒子
- 每隔几 tick 动态刷新边框
- 在四角显示更明显的竖线或光柱感
- 给玩家显示 ActionBar 提示当前操作状态

## 2.3 可视化选区优化

当前创建默认只能是“当前单区块创建”。

如果后续想更强，可以做成：

- 玩家先选起点区块
- 再选终点区块
- 自动形成矩形区块范围
- 然后再确认创建

这会更接近“真正的圈地体验”。

但这样复杂度会明显上升。

## 2.4 菜单样式统一优化

当前菜单已经统一成玻璃板背景，但仍然只是功能型界面。

后续可以继续优化：

- 统一按钮图标语义
- 为不同操作增加颜色层级
- 首页增加玩家领地数量、当前价格档位等摘要信息
- 列表页支持分页
- 详情页显示更多信息，如总区块数、创建时间、价格档位

## 2.5 与服务器主菜单整合

当前比较现实的优化方案不是替换 Java GUI，而是保留本插件作为业务入口：

- 外部主菜单只放一个“领地”入口按钮
- 按钮调用 `/mmmland menu`
- 核心状态、权限、扣费、确认流程仍由本插件处理

这样能兼顾灵活性和维护成本。

## 2.6 价格模型增强

当前价格体系已经能用，但还比较基础。

可扩展方向：

- 不同世界使用不同价格
- 不同权限组使用不同价格倍率
- 不同领地面积区间使用不同扩张单价
- 节日活动折扣
- 首次圈地免费但必须满足某些条件

## 2.7 领地限制规则增强

当前规则主要有：

- 世界限制
- 中心保护半径
- 数量限制
- 最小区块限制

后续可以增加：

- 某些世界只能在指定区域圈地
- 某些世界禁止扩张但允许创建
- 某些玩家组只能拥有特定大小范围
- 强制相邻圈地才能扩张
- 强制与他人领地保持最小间距

## 2.8 管理与审计能力增强

对于服务器长期运营，建议后续加上：

- 领地操作日志
- 创建/扩张/缩小/删除记录
- 管理员查询玩家领地统计
- 管理员查看玩家累计花费
- 异常操作报警

## 2.9 配置热重载能力

当前没有完整的配置重载管理能力。

建议后续增加：

```text
/mmmland reload
```

用于：

- 重读 `config.yml`
- 重载消息文本
- 重载价格与限制规则

## 2.10 对 Residence 依赖的健壮性增强

当前本插件强依赖 `Residence` 正常启动。

建议进一步增强：

- 启动时输出更明确的依赖检查日志
- 如果 `Residence` 未启用，给出更清晰的错误原因
- 增加启动自检信息，说明当前已连接到哪几个依赖

## 3. 当前实现层面的技术优化建议

## 3.1 把确认状态从 GuiService 拆出去

当前创建确认逻辑在 `GuiService` 中。

短期能用，但长期会让 GUI 类承担过多职责。

建议新增类似：

- `PendingActionService`
- `ClaimCreationSessionService`

专门处理：

- 待确认状态
- 会话超时
- 聊天确认
- 越界取消

## 3.2 增加分页和搜索

当前领地列表如果很多，会受菜单格子限制。

建议增加：

- 上一页 / 下一页
- 按名称搜索
- 按世界筛选

## 3.3 增加更稳定的消息与国际化结构

当前所有文本都在一个 `config.yml` 里。

后续建议拆成：

- `config.yml` 放业务配置
- `lang/zh_CN.yml` 放文本

这样更清晰，也方便多语言。

## 3.4 增加测试与模拟校验

当前主要靠实机测试。

后续建议至少把纯逻辑部分抽出来，便于测试：

- 价格计算
- 权限上限计算
- 区块边界扩张与缩小
- 中心保护区判定

## 3.5 外部入口配置

当前不再自动导出外部菜单配置。

后续可以考虑：

- 在服务器主菜单中手动增加 `/mmmland menu` 入口
- 在说明菜单中展示领地基础规则
- 通过 PlaceholderAPI 展示领地数量、上限、价格和选区状态

## 4. 建议的后续开发优先级

建议按下面顺序推进：

1. 完善创建会话流程
2. 完善重命名的菜单化输入
3. 给扩张/缩小也加确认逻辑
4. 增加管理员工具命令
5. 增加配置重载
6. 增加历史 Residence 领地导入工具
7. 视需求决定是否接入服务器主菜单入口

## 5. 总结

当前插件已经从“概念验证”进入“可用雏形”阶段。

它已经能完成核心业务，但离“正式长期服可维护版本”还有几块需要补：

- 更完整的会话交互
- 更强的管理员工具
- 更好的数据自检
- 更合理的结构拆分
- 更深的菜单整合

如果继续推进，建议下一阶段重点不是继续堆新功能，而是先把“交互流程统一”和“管理能力补齐”做好。这样后续扩展会稳很多。
